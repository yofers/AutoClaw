## Design Context

### Users
Primary users are developers or technically capable operators managing a local OpenClaw installation. They use this interface as a private control plane to install OpenClaw, initialize it, monitor gateway state, inspect command output, and edit configuration without dropping into the terminal for every task.

### Brand Personality
Calm, precise, operational.

The interface should feel dependable and controlled rather than flashy. It should reduce ambiguity, make system state legible, and convey that actions are local, direct, and reversible.

### Aesthetic Direction
Build a minimal control-panel UI for a local operations surface. Favor a clean light theme, restrained color, sharp information hierarchy, and a composed editorial feel over dashboard clichés. The product should look like a thoughtful operator console, not a startup landing page and not a generic dark hacker UI.

Avoid:
- loud gradients as the primary visual idea
- glassmorphism
- neon/dark “terminal aesthetic”
- oversized hero marketing layouts
- card grids that flatten operational hierarchy

### Design Principles
1. Show operational state first. Users should immediately understand install status, gateway health, version state, and the next safe action.
2. Keep actions obvious and compact. Installation, onboarding, service control, and config editing should feel deliberate, not hidden behind visual flourish.
3. Use restraint for trust. Minimal color, crisp spacing, and strong typography should make the interface feel reliable.
4. Treat logs and config as first-class surfaces. Console output and configuration editing are core workflows, not secondary utilities.
5. Design for repeat visits. This is a working admin surface, so the UI should stay readable during frequent use on desktop and mobile.
